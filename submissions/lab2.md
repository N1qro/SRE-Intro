### 2.6: Service Discovery Mechanism

**How the gateway finds the events service:**

The gateway discovers the events service through Docker Compose's **embedded DNS server** at `127.0.0.11:53`. When the gateway makes an HTTP request to `http://events:8081/health`:

1. The resolver at 127.0.0.11 intercepts the DNS query for "events"
2. Docker's embedded DNS responds with the **assigned IP address 172.25.0.5** (the address of app-events-1 on the app_default bridge network)
3. The gateway establishes a connection to 172.25.0.5:8081
4. Service discovery succeeds because Docker Compose automatically registers each service's name in the network's DNS resolver

**The "events" hostname resolves to 172.25.0.5**, the container's internal IP address on the app_default bridge network. This works because:
- Each service joins the same `app_default` network (defined in docker-compose.yaml)
- Docker's embedded DNS server maintains a DNS zone for that network with entries like `events.app_default: 172.25.0.5`
- Containers can resolve service names by DNS, without needing IP addresses to be hardcoded

### Annotated Request Flow

**Step 1: Gateway receives client request (14:42:48.389)**
- Client POSTs to gateway: `POST /events/1/reserve`
- Gateway logs outgoing request to events

**Step 2: Events reserves tickets (14:42:48.378-14:42:48.383)**
- Gateway forwards request to events at 172.25.0.5:8081
- Events service processes: creates reservation_id `174c25db-4deb-4d9a-b4a9-3e217de4ff64`
- Events logs: "Reserved 1 tickets for event 1"
- Response sent back to gateway

**Step 3: Gateway confirms reservation to client (14:42:48.390)**
- Gateway receives 200 OK from events
- Gateway returns reservation details to client including reservation_id

**Step 4: Client initiates payment (14:42:48.529)**
- Client POSTs to gateway: `POST /reserve/{reservation_id}/pay`
- Gateway forwards payment request to payments service

**Step 5: Payments processes charge (14:42:48.525-14:42:48.526)**
- Payments service processes: generates payment_ref `PAY-A78BAAAE`
- Payments logs: "Payment success"
- 200 OK response sent to gateway

**Step 6: Gateway confirms payment with events (14:42:48.600)**
- Gateway forwards confirmation request to events: `POST /reservations/{reservation_id}/confirm`
- Events marks reservation as confirmed, moves to orders table

**Step 7: Gateway returns final response to client (14:42:48.601)**
- Gateway receives 200 OK from events confirmation
- Gateway returns order confirmation details to client

### End-to-End Timing Analysis

| Event | Timestamp | Service | Duration |
|-------|-----------|---------|----------|
| Client sends reserve request | 14:42:48.389 | gateway | Start |
| Events processes reservation | 14:42:48.378-14:42:48.383 | events | 5ms |
| Gateway returns reserve response | 14:42:48.390 | gateway | +1ms (total 11ms from start) |
| Client sends pay request | 14:42:48.529 | gateway | +139ms (new request) |
| Payments processes charge | 14:42:48.525-14:42:48.526 | payments | 1ms |
| Events confirms order | 14:42:48.598-14:42:48.599 | events | 1ms |
| Gateway returns pay response | 14:42:48.601 | gateway | +1ms (total 72ms from pay request start) |

**Total end-to-end time:**
- **Reserve request:** 11ms (from gateway receive to response) — dominated by events service processing time
- **Pay request:** 72ms (from gateway receive to final response) — includes payments processing (1ms) + events confirmation (1ms) + round-trip latency

**Complete flow (client perspective):** From first request to final confirmation = ~11ms + 72ms = **~83ms total** including both reserve and pay operations (excluding time spent by client between the two requests).
